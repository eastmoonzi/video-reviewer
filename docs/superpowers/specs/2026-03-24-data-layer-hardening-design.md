# 视频审查工作台 — 数据层加固设计

**日期**：2026-03-24
**状态**：brainstorming 进行中（待提供示例数据后继续）

## 背景

视频审查工作台目前供团队内数人使用，审核视频模型的结构化输出。目标是提升工具可靠性后支撑更大团队使用。

### 当前核心痛点

1. **JSON 解析容错不完备** — 模型的结构化输出不稳定，JSON 格式错误导致导入解析失败。脚本容错逻辑永远补不完。现有 LLM 修复方案效果不错但需手动触发。
2. **原始文本存储不稳定** — 存储"原始文本"的逻辑有 bug，导致原文不完整时从根本上无法修复。
3. **三种模式格式碎片化** — segment / profile / audiovisual 三种审核模式各有独立的导入格式、数据流和渲染逻辑，一次改动难以覆盖全部。
4. **localStorage 状态脆弱** — 无版本管理，状态残留导致幽灵 bug（如评分面板消失）。

### 次要痛点（已记录，暂不在本次范围内）

- **部署困境** — 本地 HTML 无法推送更新，GitHub Pages 无法播放内网 HTTP 视频。无内部服务器可用。
- **数据协作** — 多人各自审核后需线下汇总。目前可接受"方便导入/导出/合并"即可，不需要实时同步。

## 选定方向：数据层加固

从三个候选方向中选定：

| 方向 | 内容 | 结论 |
|------|------|------|
| **A. 数据层加固** | 统一数据模型、原文完整保存、LLM 修复自动化、状态版本化 | **选定** |
| B. 协作与分发 | 导出标准化、内网部署方案、版本提示 | 暂缓（基础不牢上层不稳） |
| C. 审核效率 | 批量操作、撤销/重做、快捷键 | 暂缓（锦上添花） |

## 改进要点（初步）

### 1. 统一数据模型

三种审核模式共用一套导入/存储/导出管道，模式差异只在渲染层处理，消除三套独立数据流。

### 2. 原始文本完整保存

导入时第一步：完整存储原始文本（不做任何解析）。结构化解析在原文之上进行，解析失败不影响原文完整性。

### 3. LLM 修复自动化

解析失败的条目自动排队修复，而非手动逐条触发。修复结果缓存，避免重复调用。

### 4. localStorage 状态版本化

为存储数据添加版本号。版本不匹配时自动迁移或清理，防止状态残留导致 UI 异常。
